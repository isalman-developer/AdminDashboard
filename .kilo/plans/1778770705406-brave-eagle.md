# Optimized Settings System Plan

## Goal
Store settings in DB, move cache to Redis, create easy-to-use helper for single-value access in Blade and classes.

---

## Current State
- Settings stored in `settings` table with unique key
- Repository pattern with per-key caching (1hr TTL)
- Cache driver: `database` (still hits DB)
- Observer clears cache on changes
- No easy global accessor (must resolve service)
- No need for bulk operations (all keys, all settings) at this time

---

### Step 2: Optimize SettingRepository

**2.1 Add in-memory deduplication**
Add static property `$loaded = []` to store values within single request.

**2.2 Optimize single-key queries**
- Use `where('key', $key)->value('value')` for direct scalar retrieval
- Only fetch the `value` column (not entire model)
- Store result in Redis with 1-hour TTL (safety net)
- Also store in in-memory array for duplicate calls within same request

**2.3 Optimize multiple-key queries (getMultiple)**
- Check in-memory first for all keys
- Use Redis `getMultiple()` for remaining keys (single round-trip)
- Fall back to single `WHERE IN` DB query for cache misses
- Populate both Redis and in-memory after DB query

**2.4 Set operation: eager cache warming**
After `set()`, immediately update Redis + in-memory so next read is instant.

**2.5 Delete operation: clear both caches**
Remove from Redis and in-memory immediately.

**REMOVED** (not needed):
- `allKeys()` method (fetching all setting keys)
- `all()` method (fetching all settings)
- `preloadAll()` method (preloading everything)

### Step 3: Create Global Helper Functions

**3.1 Create `app/helpers.php`**
```php
<?php

use App\Services\SettingService;

/**
 * Get a single setting value.
 * Uses 3-tier cache: in-memory → Redis → DB.
 *
 * @param string $key     Setting key
 * @param mixed  $default Default value if not found
 * @return mixed
 *
 * @example setting('site_name', 'Default Site')
 * @example setting('items_per_page', 25)
 */
function setting(string $key, mixed $default = null): mixed
{
    static $service = null;

    if ($service === null) {
        $service = app(SettingService::class);
    }

    return $service->get($key, $default);
}

/**
 * Get multiple specific settings at once.
 * More efficient than calling setting() multiple times.
 *
 * @param array $keys Array of setting keys to fetch
 * @return array<string, mixed> Associative array [key => value]
 *
 * @example settings(['site_name', 'admin_email', 'items_per_page'])
 */
function settings(array $keys): array
{
    static $service = null;

    if ($service === null) {
        $service = app(SettingService::class);
    }

    return $service->getMultiple($keys);
}
```

**Why two helpers?**
- `setting('key')` → single key, simplest
- `settings(['a','b','c'])` → batch fetch, avoids N Redis calls (uses 1 MGET + 1 DB query max)

```json
}
```

## Simplified Architecture

```
setting('key')
  → SettingService::get()
    → SettingRepository::get()
      ├─ 1. Check static $loaded (in-memory array, per-request)
      ├─ 2. Check Redis: Cache::get('setting.key')
      └─ 3. DB: Setting::where('key',$key)->value('value')
           → Store in Redis (TTL: 1hr) → Store in $loaded → return
```

**Cache hierarchy:**
| Layer | Storage | Scope | Speed | TTL |
|-------|---------|-------|-------|-----|
| In-memory `$loaded` | PHP process memory | Single HTTP request | ~0ms | Request |
| Redis `setting.*` | RAM (Redis DB 1) | App-wide shared | ~2-5ms | 1 hour |
| Database (`settings`) | Disk/SSD | Persistent source | ~20-80ms | N/A |

**Removed complexity (not needed):**
- ❌ `allKeys()` - fetching all setting keys
- ❌ `all()` - fetching all settings at once
- ❌ `preloadAll()` - preloading everything into cache
- ❌ `setting.all_keys` cache key
- ❌ `setting.all` cache key
- ❌ `forever()` cache - replaced with 1-hour TTL

---

## Expected Performance Gains

| Operation | Cold (first access) | Warm (Redis) | Same request duplicate |
|-----------|---------------------|--------------|------------------------|
| `setting('key')` | ~60ms (DB + Redis write) | ~5ms (Redis read) | ~0ms (memory) |
| `settings(['a','b','c'])` | ~60ms (1 DB query for all 3) | ~5ms (1 Redis MGET) | ~0ms (memory) |

**VS old system** (database cache):
- Old cold: ~50ms (DB query with database cache table)
- Old warm: ~15ms (database cache table read)
- Old duplicate: ~15ms (cache hit still hits DB)

**Improvement**: Redis is ~3-10× faster than database cache table.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Redis not installed | Fallback to `array` cache driver for local (no persistence needed) |
| Helper function name conflict | Use unique prefix if needed (`app_setting()`, `get_setting()`) |
| Static memory per request | Limited to keys actually accessed; cleared after request |
| Redis connection failure | Laravel cache layer handles gracefully; falls back to DB |
| Stale cache if observer fails | TTL (1 hour) ensures eventual refresh; observer handles immediate invalidation |

---

## Alternatives Considered

| Alternative | Why Not Chosen |
|-------------|----------------|
| Facade `Setting::get()` | Helper is more concise, no class needed |
| Config files for everything | Admin needs to edit without deploys |
| Cache tags | Redis required; current per-key is simpler |
| View composer injection | Overhead on every request; helper lazy-loads |
